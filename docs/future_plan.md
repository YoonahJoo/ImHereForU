# ImHereForU — 미래 설계 & 확장 로드맵

> **이 문서의 목적**
> 발표 슬라이드(9번 "미래 설계 & 수익화")에서 다 담지 못한 **정확한 코드 근거와 확장 설계**를 정리한 문서다.
> 핵심 원칙: **추측이 아니라 현재 코드 구조 위에서** "무엇을 바꾸고, 어디서 백엔드가 등장하며, 무엇이 아직 미해결인지"를 정직하게 적는다.
> 한 줄 요약: **MVP는 의도적으로 프론트로 완결했고, 확장의 모든 길에 백엔드가 언제·왜 필요한지 알고 설계해뒀다. — 프론트에서 시작해 풀스택으로 끌고 갈 계획.**

---

## 0. 현재 아키텍처 (확장의 출발점)

| 항목 | 현재 상태 | 코드 근거 |
|---|---|---|
| 실행 형태 | Electron + React (서버 없음, localStorage만) | `electron/`, `src/` |
| 창 구조 | 책 창 + 바탕화면 오버레이 창 (2개) | `electron/main.ts` |
| 창 간 통신 | 메인 프로세스가 **IPC**로 중계 | `electron/main.ts`, `electron/preload.ts` |
| 캐릭터 그림 | 표정별 PNG를 **Record로 매핑** | `src/components/Character/Character.tsx` (`CHARACTER_IMAGES`) |
| 캐릭터 정체성 | '나(윤아)' 하나로 고정 | `src/assets/character/*.png` |
| 말풍선 문구 | 정적 배열 | `src/data/messages.ts` |
| 저장소 | localStorage (gifts, settings) | `src/utils/storage.ts` |
| 서버 / 계정 | 없음 (네트워크 요청 코드 0줄) | 전체 |

> *현재 캐릭터의 통짜 PNG 12장 구조는, AI 이미지 생성이 **매 생성마다 동일 캐릭터를 재현하지 못하고** 리깅용 레이어를 깨끗이 분리해주지 못하는 한계를 **직접 확인하고 택한 의도적 설계**다. (하드코딩이 아니라, 지금 AI가 안정적으로 줄 수 있는 형식에 맞춘 판단)*

**핵심:** 지금은 "내 PC 안에서 혼자 도는 앱"이다. 데이터가 PC 밖으로 나가지 않으므로 운영비가 0이다.
→ 그래서 **"두 사람을 잇는 기능"부터 반드시 백엔드가 필요**하고, 그 지점이 그대로 유료화의 경계가 된다.

## 1. 확장 로드맵 3단계

각 단계에 **비전 / [프론트=내가] / [백엔드=내가 구축] / 기존 코드 근거**를 명시한다.

### 【1단계·핵심】 둘을 잇기 = 백엔드 진입

> **비전:** 커플이 계정으로 서로를 파트너로 연결 → 편지 → 실시간 대화

**파트너 매칭 → 편지 → 실시간 대화**

- **매칭:** 한쪽이 초대 코드 생성 → 상대 입력 → 서버가 커플 ID로 묶어 DB 저장
- **편지:** 한쪽이 쓴 편지를 상대가 **Focus 일정 시간 성공 시 열람**
- **실시간 대화:** 입력 중이면 상대 캐릭터 위 "…", 전송하면 그 말이 상대 캐릭터 말풍선으로

**★ 망상이 아니라 기존 코드 위에 쌓는 설계 (검증됨)**

- **편지:** 집중 완료 신호가 `useFocusTimer`의 `onComplete` → `YoonahRoom.tsx`의 `handleFocusComplete`로 이어져, 이미 **"완료 → 보상 지급(localStorage 저장) → 상황별 창 라우팅(책 안/데스크탑)"** 파이프라인이 작동 중이다. 편지는 이 자리의 **"랜덤 선물"을 "서버가 내려주는 상대 편지"로 교체**하는 것. (열람 조건 판정·누적은 서버가 담당해 조작 방지)
- **실시간:** `OverlayApp.tsx`가 **이미 외부 이벤트(IPC 신호)를 받아 `showBubble` / `setExpression`을 호출**하고 있다. 즉 "외부 신호 → 캐릭터 반응" 구조가 이미 돌아가며, 확장은 그 **이벤트 소스를 IPC → WebSocket으로 교체**하는 것이다.

**[프론트]** WebSocket 클라이언트, 상태 동기화, 편지/매칭 UI
**[백엔드]** 계정·인증, DB(커플 관계·편지), 편지 열람 **조건 판정·누적**(서버측이라 클라 조작 불가)

---

### 【2단계】 캐릭터를 '나' → 사용자별 원하는 캐릭터로

> **비전:** 사용자가 파트너 닮은 캐릭터를 갖게 한다. 사용자는 **버튼만** 누른다.

- **관건은 인터랙션이 아니라 "동일 캐릭터의 일관된 생성"이다.** 클릭·드래그·표정 전환 엔진은 그림 출처와 무관하게 **이미 통짜 PNG 세트 위에서 작동**한다.
- 앱은 이미 **표정 → 이미지 Record 구조**(`Character.tsx`)라, **매핑 출처만 바꾸면 어떤 캐릭터든 수용**한다. 12표정 슬롯 계약은 `src/types/index.ts`의 `Expression` 타입에 이미 정의돼 있다.

**[프론트]**
- 커스텀 UI: 사진 업로드 · 가이드 촬영 틀 · 미리보기 · 확정 · 재생성
- 클라이언트 입력 검증 (얼굴 검출·해상도·블러) — 나쁜 입력은 서버로 보내기 전에 반려
- 통합/상태관리: 로딩·에러·빈 상태, 캐릭터 교체

**[백엔드]**
1. **업로드·저장:** 사진 업로드 API + 오브젝트 스토리지(S3 등)
2. **서버 검증:** 얼굴검출·해상도 재확인 (클라 검증만 믿지 않는 신뢰 경계)
3. **생성 오케스트레이션:** 생성 API(fal.ai·Replicate)로 미세학습 실행 → **12개 표정 지정 생성** *(모델을 직접 학습하지 않고 서비스 호출로 실행)*
4. **작업 큐 + 상태:** 생성이 수 분 걸리므로 **비동기 큐** + 진행상태 폴링/웹훅
5. **후처리:** 배경 제거(rembg 등 전용 모델) · 정렬 → CDN 업로드
6. **저장·인증:** `{ 표정: 이미지주소 }` 매핑을 DB에 사용자별 저장, 계정으로 소유권 관리 (1단계 인프라 공유)

> **왜 GPT/DALL·E가 아니라 Stable Diffusion인가:** LoRA·DreamBooth는 모델 내부(가중치) 접근이 필요한 미세학습 기법이라, 가중치가 공개된 **Stable Diffusion** 계열에서만 가능하다. GPT/DALL·E는 닫혀 있어 학습 불가. → "프롬프트로 GPT에 부탁"은 일관성 문제의 예시일 뿐, 실제 생성 엔진은 SD + LoRA다.

---

### 【3단계·과제】 더 활발한 인터랙션 (정직하게 미해결)

> **비전:** 부위가 따로 움직이는 "살아있는" 캐릭터. 상대와 상호작용한다고 느끼게 하는 가장 큰 요소.

- 현재는 통짜 PNG라 부위가 따로 못 움직인다. 최선의 방향은 **부위 분리 + Rive/Live2D** 리깅.
- *부분 애니의 씨앗은 이미 코드에 있다:* 눈 깜빡임을 **레이어 겹침**으로 처리 중(`Character.tsx`의 blink overlay).
- **정직한 현황:** 사용자별 에셋 **자동 분리**는 "원클릭 완성형"이 아직 업계에도 없어 **미해결 과제로 인지 중**. 현실 후보는 부위 분리 + 수작업 리깅이지만, 사용자별 자동화가 관건이다.

**[프론트]** Rive 상태머신(클릭·상태에 따라 애니가 바뀌는 구조) 연동·인터랙션 설계
**[백엔드]** 리그 에셋(`.riv`/Live2D 모델) 스토리지·CDN 서빙 / *(연구)* SAM 같은 세그멘테이션으로 레이어 자동 분리 시도 = **바로 이 지점이 미해결 핵심**
> *3단계는 본질이 프론트(인터랙션)·콘텐츠라 백엔드 비중이 얇다. 과장하지 않는다.*
>
> *커스텀(2단계 = 어떤 캐릭터냐)과 리깅(3단계 = 어떻게 움직이냐)은 별개 과제지만, 부위 분리가 되면 커스텀이 "부품 교체"로 바뀌어 둘이 수렴한다. 그래서 순서상 리깅 기반을 먼저 고려한다.*

## 2. 캐릭터 생성 — 실패 모드와 가드레일

"아무 사진 → 완벽한 캐릭터"가 아니다. **실패를 관리하는 파이프라인**이라는 점이 이 설계의 핵심이다.

| 예상 허점 | 원인 | 가드레일 |
|---|---|---|
| 배경이 바둑판처럼 남음 | 생성 AI에 "투명"을 맡기면 투명을 *그려버림* | 생성은 **단색 배경**으로 → 배경제거는 **전용 모델(rembg/remove.bg/SAM)**로 분리 |
| 얼굴 인식 실패·저품질 | 흐림·옆모습·저해상도 입력 | **생성 전 입력 게이트**(얼굴검출·해상도·블러) + **가이드 촬영 UX** + 여러 장 요구 |
| 표정 일관성 흔들림 | LoRA도 100% 아님 | 슬롯당 다중 생성 → 자동 필터 / **"재생성" 버튼** |
| 어떤 PNG가 어떤 표정? | — | **표정별 지정 생성** → 나온 순간 라벨 확정 (분류·판단 불필요) |

## 3. 수익화 = 커플 구독제

| | 무료 (혼자) | 유료 (둘을 잇기) |
|---|---|---|
| 비용 | 운영비 0 | 서버 비용 발생 |
| 기능 | 내 캐릭터 동반 | 매칭 · 편지 · 실시간 대화 |

**결정 한 줄:** 혼자 외로움을 달래는 부분은 데이터가 PC를 안 떠나 운영비가 0이라 무료, **둘을 실시간으로 잇는 순간부터 계정·DB·실시간 연결이 24시간 돌아 서버 비용이 발생하니 유료.** 이 **기술 비용의 경계가 그대로 과금의 명분**이 된다.

**구축 순서:** 프론트 확장(인터랙션) → 커플 토대(계정·DB) → 편지 → 실시간. *비용·난이도순.*

## 4. 예상 질문 Q&A (면담 대비)

1. **둘을 어떻게 커플로 연결?** → 초대 코드 생성 → 상대 입력 → 서버가 커플 ID로 DB 저장. (계정·DB부터 구축)
2. **편지 "1시간 열람" 조작 방지?** → 열람 조건 판정을 프론트가 아닌 **서버**가 함. 완료 신호는 이미 `useFocusTimer.onComplete`에 존재.
3. **실시간이 왜 일반 통신으론 안 됨?** → 매번 새로 연결해 느림. 연결을 계속 여는 **WebSocket**으로 즉시 양방향. 표시 함수(`showBubble`/`setExpression`)는 이미 외부 이벤트(IPC)로 호출 중 → 소스만 교체.
4. **AI로 캐릭터 뽑나요? GPT?** → 아니요. 일관 생성은 **Stable Diffusion + LoRA**(가중치가 열린 모델)에서 하고, 저는 모델을 직접 학습하는 게 아니라 **생성 API를 호출·오케스트레이션**하고 그 주위 백엔드(업로드·큐·후처리·저장)를 구축합니다.
5. **생성된 그림을 코드 없이 어떻게 앱에 반영?** → 사용자는 "확정"만. 앱은 이미 **표정→이미지 매핑 구조**라 매핑 출처만 바꾸면 캐릭터가 교체됩니다.
6. **이상한 사진 넣으면?** → **생성 전 입력 게이트**(얼굴검출·해상도·블러)로 반려·재촬영 유도, 배경은 생성기가 아닌 **전용 모델**로 제거합니다.
7. **캐릭터가 더 살아있게는?** → 부위 분리 + Rive가 방향이지만, **사용자별 자동 분리는 아직 미해결**로 인지하고 있습니다. (정직)

## 5. 코드 레퍼런스 색인 (근거 파일)

| 확장 | 근거 코드 | 역할 |
|---|---|---|
| 편지 | `src/hooks/useFocusTimer.ts` (`onComplete`) → `src/components/YoonahRoom/YoonahRoom.tsx` (`handleFocusComplete`) | 완료 → 보상 → 창 라우팅 파이프라인 |
| 실시간 | `src/overlay/OverlayApp.tsx` (IPC 구독 → `showBubble`/`setExpression`) | 외부 신호 → 캐릭터 반응 구조 |
| 캐릭터 교체 | `src/components/Character/Character.tsx` (`CHARACTER_IMAGES` Record) | 표정→이미지 매핑, 출처 교체 지점 |
| 표정 슬롯 계약 | `src/types/index.ts` (`Expression`) | 12표정 키 정의 |
| 날씨(무료 예시) | `src/data/messages.ts` (`timeBasedMessages`) | 시간대 멘트 패턴에 날씨 멘트 확장 |
| 부분 애니 씨앗 | `src/components/Character/Character.tsx` (blink overlay) | 레이어 겹침으로 눈 깜빡임 처리 |

> **주의:** 타이머 로직은 `src/components/FocusMode/FocusMode.tsx`(빈 파일)가 아니라 **`useFocusTimer` 훅 + `YoonahRoom.tsx`**에 있다.
