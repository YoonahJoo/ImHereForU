## v1.0 — MVP Design (2026-05-24)

### 앱 개요

Electron 기반 데스크탑 동반자 앱. 항상 화면 위에 떠 있는 작은 캐릭터 윈도우.  
창 크기: 320 × 480px (고정)

---

### 캐릭터 스타일

- Blythe 돌 / chibi 일러스트 감성
- 큰 검은 눈(하이라이트 있음), 긴 속눈썹, 홍조, 주근깨
- 연필/크레용 텍스처 느낌의 손그림 스타일
- 현재 이모지로 표현 중 (ex: 😚🥰😪👩‍💻)
- 향후 일러스트 에셋으로 교체 예정

---

### 컬러 팔레트

| 역할 | 색상 | 헥스 |
|------|------|------|
| Primary | Pastel Pink | `#FFB7C5` |
| Secondary | Pastel Blue | `#B8D4F0` |
| Accent | Pastel Yellow | `#FFF0C8` |
| Background | Cream Pink | `#FFF5F7` |
| Text | Muted Purple-Brown | `#6B5B73` |
| Border / Line | Dusty Rose | `#E8A0B0` |

---

### UI 무드 키워드

kawaii / stationery / Sanrio / soft / cozy / 귀여운 데스크탑 위젯

---

### 디자인 원칙

1. 모든 테두리는 scalloped(물결) 또는 rounded — 날카로운 직각 없음
2. 배경에 subtle한 격자(grid) 또는 깅엄(gingham) 패턴
3. 장식 요소: 리본, 별, 하트, 반짝이 — 과하지 않게 포인트로
4. 버튼은 파스텔 계열 fill + 얇은 dusty rose 테두리
5. 폰트: Nunito / Quicksand / rounded sans-serif 계열

## v2.0 — 책 세계관 리워크 (2026-06-15 ~ 2026-06-22) ✅

### 1. MVP 디자인의 한계 — 왜 바꿔야 했는가

이 앱의 핵심 가치는 PRD에서 **Emotional Presence**로 정의된다.

> "사용자가 혼자 작업하는 시간에도 상대방의 존재감을 느낄 수 있도록 돕는다."

MVP를 완성한 후 이 기준으로 디자인을 다시 봤을 때, 두 가지 문제가 명확했다.

**문제 1 — 캐릭터가 보이지 않았다**  
표정 시스템, 인터랙션 로직, 말풍선까지 기능적으로는 완성됐지만, 캐릭터 자체가 이모지 하나로 대체되어 있었다. "상대방의 존재감"을 전달하는 앱에서 정작 상대방이 보이지 않았다.

**문제 2 — UI가 앱의 감정적 목적을 전달하지 못했다**  
흰 배경의 div에 파스텔 팔레트를 입힌 UI는 일반적인 생산성 위젯과 구분되지 않았다. 캐릭터가 "사는 공간"이라는 감각이 없었고, 사용자가 앱을 열었을 때 "누군가의 세계에 접속한다"는 감각도 없었다. 기능은 동작했지만 감성이 전달되지 않았다.

---

### 2. 새로운 컨셉 — 책, 그리고 존재감의 단계적 확장

MVP의 문제를 해결하는 방향을 찾는 과정에서, 단순히 "예쁜 배경"을 추가하는 것으로는 부족하다는 판단을 했다. 디자인이 앱의 서사와 맞닿아 있어야 했다.

이 앱의 최종 목표는 캐릭터가 책에서 나와 사용자의 데스크탑 위에서 독립적으로 살아 움직이는 것(v3.0)이다. 그 여정을 역방향으로 생각했을 때, 단계가 보였다.

```
v1.0  캐릭터가 인터페이스 안에 있다      presence in an interface
v2.0  캐릭터가 자신만의 세계에 있다      presence in their own world
v3.0  캐릭터가 사용자의 세계로 나온다    presence in your world
```

이 서사에서 **책**은 두 가지 역할을 동시에 수행한다.

**역할 1 — 캐릭터의 세계를 만든다**  
책은 캐릭터가 살고 있는 공간을 시각적으로 정의한다. 사용자가 앱을 열 때 책을 펼치는 행위 자체가 "캐릭터의 세계를 방문한다"는 의미가 된다. 공간이 생기면 캐릭터의 존재감도 생긴다.

**역할 2 — v3.0의 논리적 전제가 된다**  
캐릭터가 "책 밖으로 나온다"는 개념은 먼저 책이라는 세계가 확립되어 있어야 성립한다. v2.0에서 책을 캐릭터의 원점으로 만들어 두면, v3.0에서 그 세계를 떠나 사용자의 데스크탑으로 나오는 순간이 극적으로 느껴진다. **책은 목적지가 아니라 출발점이다.**

또한 책/스테이셔너리 무드는 이 앱의 주요 사용 맥락인 "공부하면서 함께 있는 감각"과 자연스럽게 연결된다.

### 3. 주요 디자인 결정

#### 3-1. 투명 프레임리스 윈도우

| 항목 | v1.0 | v2.0 |
|---|---|---|
| 윈도우 스타일 | 기본 OS 창 테두리 | `transparent: true, frame: false` |
| 배경 | 흰 div | 투명 — 책 PNG만 화면에 떠 있음 |

책이 화면 위에 물리적으로 놓인 것처럼 보이게 하려면 OS 기본 창 테두리를 없애야 했다. 투명 배경으로 책 외의 영역을 완전히 걷어냈다.

**트레이드오프**: 투명 창에서는 OS 기본 드래그가 사라진다. `-webkit-app-region: drag`로 직접 구현해야 했고, 이 설정이 mousedown 이벤트를 가로채 캐릭터의 모든 클릭 인터랙션이 작동하지 않는 문제가 생겼다. 클릭 요소 전체에 `-webkit-app-region: no-drag`를 추가해서 해결했다. **디자인 결정 하나가 인터랙션 전체를 다시 설계하게 만든 사례.**

| 항목 | v1.0 | v2.0 |
|---|---|---|
| 창 크기 | 320 × 480px | 900 × 639px |
| 비율 | 세로형 | 가로형 (책 펼친 모양) |

책을 펼친 형태는 가로형이다. 이 단순한 결정이 내부 레이아웃 전체의 재설계를 요구했다.

#### 3-3. 책 인트로 애니메이션

앱을 열면 책 표지 → 3D 플립 → 책 내지로 전환되는 애니메이션.  
구현 방식: CSS 3D flip (`transform-style: preserve-3d`, `backface-visibility: hidden`, `rotateY(-180deg)`)

**기술적 문제**: 인트로(세로형 책 표지, 500×750)와 메인 화면(가로형 책 내지, 900×639)의 창 크기가 달랐다. 하나의 Electron 창 안에서 처리할 수 없었다.  
**해결**: 애니메이션 완료 시점에 IPC로 창 크기를 동적으로 변경 (`ipcMain.on('resize-window')`). 인트로를 별도 컴포넌트(`BookIntro`)로 분리해 메인 UI와 생명주기를 독립적으로 관리했다.

#### 3-5. 캐릭터 에셋 시스템 — 레이어 방식에서 완성 이미지 방식으로

이 결정이 v2.0에서 가장 중요한 기술적 전환점이다.

**초기 계획 — 레이어 합성 방식 (폐기)**

| 레이어 | 파일 | 역할 |
|---|---|---|
| body | 1장 | 몸통 고정 |
| face | 10장 | 표정별 얼굴 |
| eyes | 2장 | 눈 뜬/감은 |
| arm | 2장 | 기본/손흔들기 |

총 15개 에셋을 CSS `position: absolute`로 적층하는 구조. 이론적으로는 레이어 조합으로 모든 표정을 만들 수 있다.

**문제 발견**: AI 이미지 생성 모델로 에셋을 뽑으면 레이어 간 픽셀 정렬이 보장되지 않는다. face 레이어와 body 레이어가 조금만 어긋나도 목이 끊기거나 얼굴 위치가 틀어진다. 수동으로 정렬을 맞추는 작업은 표정마다 반복해야 하며, 에셋이 추가될 때마다 모든 조합을 다시 검수해야 한다. **이 방식은 근본적으로 AI 생성 에셋과 맞지 않는다.**

**전환 — 완성 이미지 방식 (Method A)**

표정별로 몸 + 얼굴이 포함된 완성 이미지 1장씩. 정렬 문제가 원천 차단된다.

| 파일 | 표정 | 발생 조건 |
|---|---|---|
| `character_idle.png` | 기본 | 기본 상태 |
| `character_idle_blink.png` | 기본 (눈 감음) | idle 중 주기적 깜빡임 overlay |
| `character_wave.png` | 손인사 | 앱 시작 인삿말 |
| `character_smile.png` | 미소 | 클릭, 더블클릭, Focus 완료 |
| `character_focus.png` | 집중 (노트북) | Focus 타이머 실행 중 |
| `character_sleepy.png` | 졸림 | 15분 방치, 새벽(dawn) 시작 인삿말 후 |
| `character_sulky_daily.png` | 삐짐 | daily 모드 롱프레스 |
| `character_sulky_focus.png` | 삐짐 (노트북) | focus 모드 롱프레스 |
| `character_dragging_daily.png` | 놀람 | daily 모드 드래그 중 |
| `character_dragging_focus.png` | 놀람 | focus 모드 드래그 중 |
| `character_curious.png` | 째려봄 | focus 중 daily 전환 시도 |
| `character_cheering.png` | 미소 | focus 중 더블클릭 |

총 12개. 무브먼트(float, bounce, shake 등)는 에셋 없이 CSS keyframe으로 처리.

**에셋 생성 과정**: GPT 이미지 생성 → photoroom.com으로 배경 제거 → PNG 저장. 모든 이미지 1110×1417px 투명 PNG, 캐릭터 위치 전 이미지 동일하게 유지.

#### 3-6. 캐릭터 애니메이션 — CSS 두 레이어 crossfade

표정이 바뀔 때 이미지가 툭 끊기는 대신 부드럽게 전환되도록 두 레이어 crossfade 시스템을 구현했다.

```
layerA (normal flow, 높이 결정)
layerB (position: absolute, 위에 겹침)

activeLayer = 'A' 상태에서 B로 전환 시:
  1. layerB.src = 새 이미지
  2. layerB.opacity: 0 → 1 (0.25s)
  3. layerA.opacity: 1 → 0 (0.25s)
  4. activeLayer = 'B'
```

`activeLayerRef`로 현재 레이어를 ref에 관리해 stale closure 문제를 방지했다.

**blink는 별도 처리**: idle 중 3~5초 간격으로 `character_idle_blink.png`를 absolute overlay로 얹었다 150ms 후 제거. crossfade 없이 즉시 전환 — 눈깜빡임이 부드럽게 fade되면 오히려 어색하기 때문.

---

#### 3-7. 드래그 애니메이션 — pendulum

드래그 중 캐릭터가 기울어지는 애니메이션을 단순 `rotate(5deg)` 고정에서 **pendulum**으로 전환했다.

```css
@keyframes pendulum {
  0%, 100% { transform: rotate(-12deg); }
  50%       { transform: rotate(12deg); }
}
```

잡혀서 흔들리는 물리적 느낌이 draging_daily/focus 표정과 어울린다. 드래그 시 말풍선도 추가: `"where am I going?"` (daily) / `"Oops!"` (focus).

#### 3-8. 인터랙티브 에셋 배치 (Figma 좌표 기반)

책 배경 위에 해바라기(sunflower), 꽃다발(bouquet), 메모(memo) 등 PNG 에셋을 배치했다.  
Figma에서 `book-light.png`(900×639)를 배경으로 깔고, 각 에셋의 X/Y/rotate 값을 패널에서 읽어 CSS `position: absolute`로 그대로 적용했다.

```
sunflower (focus 모드) — x:253, y:77, rotate:175.87deg
sunflower (daily 모드) — x:190, y:457
bouquet (온보딩)       — x:711, y:410, rotate:-18.43deg
memo (gift room)       — x:565, y:44
```

**잠재적 문제**: 절대 좌표 기반 배치는 해상도가 달라질 경우 좌표가 틀어질 수 있다. 현재 단일 해상도(900×639)에 최적화된 방식이며, 향후 반응형 처리 시 재작업이 필요할 수 있다.

---

#### 3-9. 패널 오버레이 — blur+darken backdrop + floating 패널

초기 Settings / GiftRoom은 `position: absolute; inset: 0`으로 책 전체를 덮는 방식이었다. v2.0 마무리 단계에서 두 가지 문제를 발견했다.

1. 패널이 열릴 때 책 배경이 완전히 사라져 공간감이 끊긴다
2. 패널 크기가 창 전체와 동일해 "패널이 뜬다"는 감각이 없다

**변경**: 반투명 blur backdrop 위에 floating 패널이 올라오는 구조.

```
backdrop (blur 4px + rgba(0,0,0,0.35)) — 배경 흐릿하게 유지
  └── floating panel (border-radius: 20px, box-shadow)
        GiftRoom: 500×440px
        Settings: 420×480px
```

backdrop 클릭 → 닫힘, 패널 클릭 → stopPropagation. 책 세계관이 완전히 사라지지 않고 뒤에 흐릿하게 남아 있어 공간 연속성이 유지된다.

#### 3-11. 시작 인삿말 — 시간대별 분기

앱 시작 시 말풍선 3개가 순서대로 나오는 구조에 시간대별 분기를 추가했다.

| 시간대 | 시퀀스 |
|---|---|
| morning | 시간 인사 → "It's lovely to see you here" → "This is my tiny cozy space." → 온보딩 안내 |
| afternoon | 동일 |
| evening | 동일 |
| night | 기본 인사 → cozy → 온보딩 안내 → `character_sleepy` + 야간 경고 |
| dawn | 기본 인사 → cozy → 온보딩 안내 → `character_sulky_daily` + 새벽 경고 |

night/dawn은 인삿말 시퀀스 이후 시간대 맞춤 경고 메시지와 특수 표정이 추가로 출력된다. `greetingActive` ref로 시퀀스 중 시간대 메시지가 겹치는 것을 방지한다.

### 4. 기술적 문제 해결 — v2.0 전체

| 문제 | 원인 | 해결 |
|---|---|---|
| 클릭 인터랙션 안 됨 | `-webkit-app-region: drag`가 mousedown 가로챔 | 클릭 요소에 `no-drag` 추가 |
| 인트로와 메인 창 크기 충돌 | 세로/가로 전환 불가 | IPC 리사이즈로 창 크기 분리 |
| 캐릭터 레이어 정렬 문제 | AI 생성 이미지의 픽셀 정렬 불보장 | 완성 이미지 방식으로 전환 |
| 표정 전환 시 이미지 툭 끊김 | 즉시 src 교체 | 두 레이어 CSS crossfade |
| Focus pause → resume 후 표정 버그 | `mode` 비교 타이밍 이슈 | `timerStatus === 'running'` 단독 기준으로 통일 |
| 하트가 항상 같은 자리에서 뜸 | HeartEffect에 캐릭터 위치 전달 안 됨 | `offsetX/Y` prop 추가 |
| 말풍선이 드래그 시 좁아짐 | `max-width`는 컨테이너 가장자리 기준 | `width` 고정값으로 변경 |
| 시간대 인사가 인삿말과 겹침 | timeMessage가 1초 후 실행 | `greetingActive` ref 가드 추가 |

### 5. v3.0 — 캐릭터 독립 (예정)

v2.0에서 캐릭터에게 "원래 사는 세계(책)"를 부여하면, v3.0에서 그 세계를 떠나는 행위가 의미를 가진다.

v3.0의 목표: 캐릭터가 책 밖으로 나와 사용자의 데스크탑 위에서 독립적으로 움직이고 상호작용한다.

- Electron의 투명 창을 데스크탑 전체 영역으로 확장
- 캐릭터가 화면 어디서든 자유롭게 이동
- v2.0의 표정 시스템(12개 완성 이미지 + crossfade)을 그대로 재사용
- 포즈 변경 없이 기존 에셋으로 커버 가능

v2.0의 책 세계관이 확립되어 있어야 "책 밖으로 나온다"는 개념이 성립한다. v3.0은 v2.0의 논리적 귀결이다.

### 6. 배운 점 — 인터뷰용 회고

**디자인 결정이 기술 구현을 바꾼 사례들**

| 디자인 결정 | 발생한 기술적 문제 | 해결 방법 |
|---|---|---|
| 투명 프레임리스 창 | `-webkit-app-region: drag`가 클릭 이벤트를 가로챔 | 클릭 요소 전체에 `no-drag` 처리 |
| 인트로(세로)와 메인(가로) 창 크기 다름 | 하나의 창에서 처리 불가 | IPC로 창 크기 동적 변경 |
| AI 이미지로 레이어 에셋 생성 | 픽셀 정렬 보장 불가 | 완성 이미지 방식으로 전환 |
| 표정 전환을 부드럽게 | 이미지 src 교체는 항상 끊김 | 두 레이어 CSS opacity crossfade |
| 말풍선이 캐릭터를 따라다녀야 함 | 캐릭터 위치 정보가 말풍선에 전달되지 않음 | `offsetX/Y` prop 추가 |

**MVP와 v2.0의 핵심 차이**

MVP는 *기능이 동작하는가*를 검증했다.  
v2.0은 *감성이 전달되는가*를 검증했다.

같은 인터랙션(클릭, 드래그, 롱프레스)이지만, 캐릭터가 책 속 공간에 있을 때와 흰 div 안에 있을 때 사용자가 느끼는 감각이 다르다. 기능 완성도보다 **세계관의 일관성**이 이 프로젝트에서 더 중요한 디자인 지표다.

**에셋 전략에서 배운 것**

처음 레이어 방식을 선택한 건 기술적으로 "더 확장성 있어 보이는" 구조였기 때문이다. 하지만 실제 생성 도구(AI 이미지 모델)의 한계를 충분히 고려하지 않은 결정이었다. 에셋이 실제로 완성되었을 때 문제가 드러났고, 전략을 바꿔야 했다. **도구의 제약을 먼저 확인하고 구조를 결정했어야 했다.**

완성 이미지 방식으로 전환한 후에는 오히려 구현이 단순해졌다 — 레이어 정렬 코드 없이 src만 교체하면 되기 때문이다. 단순한 구조가 더 견고하다는 것을 다시 확인한 사례.
