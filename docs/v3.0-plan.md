# v3.0 기획서 — 캐릭터가 책 밖으로

**작성일**: 2026-06-22  
**상태**: 기획 (기술스택 + 구현 계획 확정)  
**선행 버전**: v2.0 책 UI 리워크 (완료)

---

## 1. 비전

지금까지 캐릭터는 **책(윈도우) 안**에서만 존재했다.  
v3.0의 목표는 캐릭터가 **책 밖, 데스크탑 위로 나오는 것** — 화면 어디든 곁에 있는 "데스크탑 메이트"로 확장한다.

> 책은 "집"이고, 캐릭터는 집 밖으로 산책 나올 수 있다.

| 항목 | v2.0 (현재) | v3.0 (목표) |
|---|---|---|
| 캐릭터 활동 범위 | 책 왼쪽 페이지 내부 | 데스크탑 전체 화면 |
| 윈도우 구조 | 책 1개 (900×639) | 책 + 캐릭터 전용 오버레이 창 |
| 상호작용 | 책 안에서 클릭/드래그 | 데스크탑 어디서든 클릭/드래그 |
| 존재감 | 앱을 열어야 보임 | 책을 닫아도 캐릭터는 화면에 남을 수 있음 |

---

## 2. 핵심 기술 과제

v3.0의 모든 기술적 복잡도는 하나의 문제에서 비롯된다.

> **"캐릭터만 클릭을 받고, 캐릭터 외 영역은 마우스 이벤트가 아래 앱으로 통과되어야 한다."**

전체 화면 크기의 투명 창을 띄워도 사용자가 그 아래의 브라우저, 터미널 등을 정상적으로 사용할 수 있어야 한다. 이 하나의 요구사항이 창 구조, IPC 설계, 이벤트 처리 방식 전반에 영향을 준다.

---

## 3. 기술 선택과 근거

### 3-1. 플랫폼: Electron 유지

**결정**: v3.0도 기존 Electron 기반을 그대로 사용한다.

**근거**:  
v3.0의 핵심 구현인 `setIgnoreMouseEvents(true, { forward: true })` — 전체 화면 창에서 특정 영역만 마우스 이벤트를 받고 나머지는 통과시키는 기능 — 은 Electron의 `BrowserWindow` API가 네이티브로 지원한다. 이 기능을 가장 적은 추가 코드로 사용할 수 있는 런타임이 Electron이다.

비교:
- **Tauri**: 동일한 per-pixel 마우스 통과 기능이 현재(2026) 안정 버전에서 미지원. 구현 가능하더라도 Rust 레이어 수정이 필요하며, 기존 React 컴포넌트 전체를 새 런타임으로 이식하는 비용이 크다.
- **NW.js**: Electron과 유사하지만 생태계 크기와 문서가 작고, `setIgnoreMouseEvents` 지원이 불안정하다.
- **네이티브 앱 (Swift/Objective-C)**: 플랫폼 종속, 기존 React 에셋·CSS 애니메이션·표정 시스템 재사용 불가.

기존 자산(Character 컴포넌트, crossfade 시스템, 말풍선, CSS 애니메이션 12종)을 **변경 없이 재사용**할 수 있다는 점도 결정적이다.

---

### 3-2. 창 구조: 두 개의 BrowserWindow

**결정**: 책 창(`bookWindow`)과 캐릭터 오버레이 창(`overlayWindow`)을 별도 `BrowserWindow`로 분리한다.

**근거**:  
하나의 창으로 처리하려면 책 창(900×639)을 화면 전체로 확장해야 한다. 이는 책 UI 레이아웃 전체를 다시 설계해야 하며, 책이 화면 가장자리에 고정되지 않고 자유롭게 이동하는 현재 동작도 깨진다.

두 창 분리의 이점:
- 책 창은 현재 크기(900×639)와 위치 자유 이동을 그대로 유지
- 오버레이 창은 화면 전체를 커버하되 완전 투명 + 마우스 이벤트 통과 설정만 담당
- 두 창의 생명주기가 독립적 → 책을 닫아도 오버레이는 유지 가능 (또는 연동 가능)
- 기존 preload.ts / IPC 구조를 그대로 재사용

```
[bookWindow]  900×639, 이동 가능, 책 + UI 전체
[overlayWindow]  화면 전체 크기, transparent, alwaysOnTop, 캐릭터만 렌더
```

---

### 3-3. 마우스 통과: setIgnoreMouseEvents + IPC 토글

**결정**: `win.setIgnoreMouseEvents(true, { forward: true })`를 기본값으로, 캐릭터 영역 hover 시 IPC로 `false`로 토글한다.

**근거**:  
Electron의 `setIgnoreMouseEvents(true, { forward: true })`는 창 전체의 마우스 이벤트를 무시하되, OS에게 이벤트를 "포워딩"해 아래 창이 정상적으로 이벤트를 받게 한다. `{ forward: true }` 없이 `true`만 사용하면 아래 앱으로 이벤트가 전달되지 않아 창 아래의 브라우저/터미널이 동작하지 않는다.

동작 원리:
1. 오버레이 창은 기본적으로 `setIgnoreMouseEvents(true, { forward: true })`
2. 렌더러에서 `mousemove` 이벤트로 커서가 캐릭터 이미지 영역 위에 있는지 감지
3. 캐릭터 위 → IPC `overlay:cursor-on-character` → 메인이 `setIgnoreMouseEvents(false)` 호출
4. 캐릭터 밖 → IPC `overlay:cursor-off-character` → 메인이 `setIgnoreMouseEvents(true, { forward: true })` 복구

대안으로 검토한 방법:
- **CSS `pointer-events: none`**: 렌더러 레벨 처리로, 실제 OS 마우스 이벤트는 여전히 창이 받음. 아래 앱으로 통과가 안 됨.
- **`screen.getCursorScreenPoint()` 폴링**: 주기적으로 커서 위치와 캐릭터 bounds를 비교하는 방식. 가능하지만 IPC 트래픽이 지속 발생하고, hover/leave 이벤트 기반보다 반응이 느림.

---

### 3-4. 창 간 상태 동기화: IPC (기존 방식 확장)

**결정**: 기존 `ipcRenderer.send` / `ipcMain.on` 패턴을 그대로 확장한다.

**근거**:  
두 창 사이에 공유해야 하는 상태는 표정(expression), 모드(daily/focus), 타이머 상태 세 가지다. 이 정도의 단방향·이벤트 기반 통신에는 Redis, SharedWorker, BroadcastChannel 등 별도 pub-sub 레이어가 불필요하다.

기존 preload.ts의 `contextBridge.exposeInMainWorld('ipcRenderer', ...)` 구조를 오버레이 창도 동일하게 사용하면, 렌더러 코드에서 `window.ipcRenderer.send(...)` 호출 방식이 동일하다. 새로운 패턴을 배울 필요 없이 기존 코드와 일관성을 유지한다.

---

### 3-5. 캐릭터 이동: CSS transform + IPC 위치 전달

**결정**: 캐릭터의 화면 위치는 전역 좌표(px)로 관리하고, `overlayWindow` 렌더러에서 CSS `transform: translate(x, y)`로 렌더링한다.

**근거**:  
WebGL, Canvas, Framer Motion 등 별도 애니메이션 엔진은 모두 오버스펙이다. 캐릭터는 2D 이미지 한 장이고, 이동 경로는 부드러운 직선 또는 곡선이다. CSS `transform: translate`는 브라우저가 GPU 가속으로 처리하며 성능이 충분하다. `transition: transform 0.3s ease-out`으로 이동 보간도 CSS 한 줄로 해결된다.

이동 로직:
- 마우스 따라다니기: `screen.getCursorScreenPoint()`를 100ms 간격으로 폴링해 목표 좌표 계산
- 느리게 따라가기: 목표 좌표와 현재 좌표의 차이에 easing 계수(0.05~0.1)를 곱해 점진적 이동 (lerp)
- 이 계산은 메인 프로세스에서 `setInterval`로 수행하거나 렌더러에서 `requestAnimationFrame`으로 처리

---

### 3-6. 상태 관리: 기존 useState/useRef 유지

**결정**: Zustand, Jotai 등 외부 상태 관리 라이브러리를 도입하지 않는다.

**근거**:  
오버레이 창의 렌더러는 단순히 Character 컴포넌트 하나를 렌더링한다. 공유해야 하는 상태는 IPC로 받아서 `useState`로 관리하면 충분하다. 상태 트리가 깊지 않고 컴포넌트 수도 적기 때문에 prop drilling 문제가 발생하지 않는다. 라이브러리 추가는 번들 크기와 학습 비용만 늘린다.

## 4. 아키텍처 설계

### 4-1. 전체 구조

```
electron/main.ts
├── bookWindow (기존, 900×639, resizable:false)
│   └── renderer: YoonahRoom (기존 코드 그대로)
│
└── overlayWindow (신규, 화면 전체, transparent, alwaysOnTop)
    └── renderer: OverlayApp
        └── Character (기존 컴포넌트 재사용)
        └── SpeechBubble (기존 컴포넌트 재사용)
```

### 4-2. overlayWindow 설정

```typescript
overlayWin = new BrowserWindow({
  x: display.bounds.x,
  y: display.bounds.y,
  width: display.bounds.width,
  height: display.bounds.height,
  transparent: true,
  frame: false,
  alwaysOnTop: true,
  focusable: false,       // 이 창이 포커스를 빼앗지 않음
  skipTaskbar: true,      // 작업 표시줄에 나타나지 않음
  webPreferences: {
    preload: path.join(__dirname, 'preload.mjs'),
  },
})
overlayWin.setIgnoreMouseEvents(true, { forward: true })
```

`focusable: false`는 캐릭터를 클릭해도 오버레이 창이 포커스를 가져가지 않아, 사용자가 작업 중인 다른 앱의 포커스가 유지된다. 단, 캐릭터와의 인터랙션(클릭, 드래그)이 필요한 순간에만 `setIgnoreMouseEvents(false)`로 이벤트를 받을 수 있다.

### 4-3. IPC 채널 설계

| 채널 | 방향 | 용도 |
|---|---|---|
| `overlay:show` | Main → Overlay | 캐릭터 표시 시작 |
| `overlay:hide` | Main → Overlay | 캐릭터 숨기기 (귀가) |
| `overlay:set-expression` | Book → Main → Overlay | 표정 동기화 |
| `overlay:set-position` | Main → Overlay | 캐릭터 위치 (x, y) 전달 |
| `overlay:cursor-on-character` | Overlay → Main | 캐릭터 위로 커서 진입 → ignoreMouseEvents 비활성 |
| `overlay:cursor-off-character` | Overlay → Main | 캐릭터 밖으로 커서 이탈 → ignoreMouseEvents 활성 |
| `book:exit-character` | Book → Main | 캐릭터 나오기 트리거 |
| `book:enter-character` | Overlay → Main → Book | 캐릭터 귀가 트리거 |
| `resize-window` | Book → Main | 기존 IPC (변경 없음) |

## 5. 핵심 구현 상세

### 5-1. 마우스 통과 토글 (메인 프로세스)

```typescript
ipcMain.on('overlay:cursor-on-character', () => {
  overlayWin?.setIgnoreMouseEvents(false)
})
ipcMain.on('overlay:cursor-off-character', () => {
  overlayWin?.setIgnoreMouseEvents(true, { forward: true })
})
```

### 5-2. 캐릭터 hit 영역 감지 (오버레이 렌더러)

`mousemove` 이벤트에서 캐릭터 이미지 엘리먼트의 `getBoundingClientRect()`와 비교:

```typescript
window.addEventListener('mousemove', (e) => {
  const rect = characterRef.current?.getBoundingClientRect()
  if (!rect) return
  const isOver = e.clientX >= rect.left && e.clientX <= rect.right
                && e.clientY >= rect.top && e.clientY <= rect.bottom
  if (isOver !== wasOver) {
    window.ipcRenderer.send(isOver ? 'overlay:cursor-on-character' : 'overlay:cursor-off-character')
    wasOver = isOver
  }
})
```

변경이 있을 때만 IPC를 보내 불필요한 메시지를 줄인다.

### 5-3. 캐릭터 이동 (부드러운 마우스 추적)

```typescript
// 렌더러 내 requestAnimationFrame 루프
let targetX = 0, targetY = 0, currentX = 0, currentY = 0
const LERP = 0.06  // 이동 속도 계수 (0에 가까울수록 느리게 따라감)

function tick() {
  currentX += (targetX - currentX) * LERP
  currentY += (targetY - currentY) * LERP
  characterEl.style.transform = `translate(${currentX}px, ${currentY}px)`
  requestAnimationFrame(tick)
}

// 메인 프로세스가 100ms 간격으로 커서 위치 전달
window.ipcRenderer.on('overlay:set-position', (_, { x, y }) => {
  targetX = x - CHARACTER_WIDTH / 2
  targetY = y - CHARACTER_HEIGHT / 2
})
```

lerp 방식은 목표 지점과의 거리에 비례해 속도가 줄어들어 자연스럽게 감속된다. 상수 속도로 이동하는 것보다 살아있는 느낌을 준다.

## 7. 단계별 마일스톤

### M1 — 오버레이 PoC (1~2일)

목표: 전체 화면 투명 창에 캐릭터가 뜨고, 캐릭터 외 영역은 클릭이 아래 앱으로 통과된다.

- [ ] `overlayWindow` 생성 (transparent, alwaysOnTop, 전체 화면)
- [ ] `OverlayApp.tsx` 작성 (Character 렌더링, idle 표정 고정)
- [ ] `setIgnoreMouseEvents(true, { forward: true })` 기본 설정
- [ ] 캐릭터 영역 hover → `setIgnoreMouseEvents(false)` 토글 확인
- [ ] 캐릭터 외 영역에서 브라우저/터미널 정상 동작 확인

**성공 기준**: 화면에 캐릭터가 뜨고, 캐릭터 옆 영역에서 다른 앱을 정상적으로 클릭할 수 있다.

---

### M2 — 책 ↔ 오버레이 상태 동기화 + 나오기 트리거 (2~3일)

목표: 책에서 트리거를 누르면 캐릭터가 오버레이에 나타난다. 표정이 두 창 사이에서 동기화된다.

- [ ] "밖으로 나오기" 트리거 UI 확정 및 구현 (예: 특정 에셋 클릭)
- [ ] 나오기/귀가 전환 애니메이션 (책 에셋에서 사라지고 오버레이에 등장)
- [ ] `overlay:set-expression` IPC 연결 (책의 표정 변경 → 오버레이 동기화)
- [ ] Focus 타이머 상태 동기화 (`isTimerRunning` 오버레이에도 전달)

**성공 기준**: 책에서 트리거를 누르면 캐릭터가 데스크탑으로 나오고, 책에서 클릭하면 오버레이 캐릭터 표정도 같이 바뀐다.

---

### M3 — 데스크탑 이동 + 행동 (3~4일)

목표: 캐릭터가 마우스를 부드럽게 따라다니고, 방치 시 대기 행동을 보인다.

- [ ] 메인 프로세스에서 100ms 간격으로 `screen.getCursorScreenPoint()` 폴링
- [ ] 오버레이 렌더러에서 lerp 기반 이동 (requestAnimationFrame)
- [ ] 마우스 추적 on/off 토글 (캐릭터를 붙잡으면 마우스 추적 중지, 드래그로 위치 이동)
- [ ] 방치 감지: 15분 이상 마우스 정지 시 sleepy 표정 + idle 행동
- [ ] 귀가 트리거: 캐릭터를 책 근처로 드래그하거나 귀가 버튼 클릭

**성공 기준**: 캐릭터가 마우스를 자연스럽게 따라다니고, 드래그로 위치를 바꿀 수 있다.

---

### M4 — 전환 연출 + 폴리시 (2~3일)

목표: 나오기/귀가 연출이 완성되고 다크/라이트 테마가 오버레이에도 적용된다.

- [ ] 책에서 나오는 연출 (책 페이지에서 캐릭터가 팝업 → 오버레이로 이어짐)
- [ ] 귀가 연출 (오버레이에서 책 방향으로 이동 후 사라짐)
- [ ] 다크/라이트 테마 동기화 (책의 테마 변경 → 오버레이 말풍선 색상 연동)
- [ ] 멀티 모니터 대응: `screen.getAllDisplays()`로 활성 디스플레이 bounds 계산

**성공 기준**: 나오기/귀가가 자연스럽고, 다크모드에서도 오버레이가 올바르게 표시된다.

## 8. 확정 필요 사항 (Open Questions)

구현 전 결정이 필요한 항목:

1. **"밖으로 나오는" 트리거** — 전용 버튼? 캐릭터를 책 밖으로 드래그? 특정 에셋 클릭?
2. **데스크탑에서의 주 행동** — 마우스 추적 vs 자유 배회 vs 한 곳에 머묾?
3. **책을 닫았을 때** — 캐릭터도 같이 사라짐 vs 데스크탑에 계속 남음?
4. **멀티 모니터** — 단일 모니터만 지원 vs 커서 따라 모니터 이동?
5. **v3.0 완료 기준** — M3까지? M4까지?
