# Git / GitHub 워크플로우 개념정리

> 바이브코딩 1달차 눈높이로 정리한, branch · commit · push · merge · 배포 개념 노트

---

## 0. 한눈에 보는 핵심

| 동작 | 한 줄 정의 | 비유 |
|---|---|---|
| **commit** | 변경사항을 로컬에 "저장 지점"으로 기록 | 게임 세이브 포인트 |
| **push** | 내 로컬 커밋을 **같은 이름의 원격 브랜치**에 업로드 | 노트를 클라우드에 백업 |
| **merge** | **A 브랜치를 B 브랜치에 합치기** | 여러 권의 노트를 한 권으로 합침 |
| **deploy(배포)** | 코드를 빌드해서 실제 서비스로 띄우기 | 원고를 인쇄해 서점에 진열 |

**가장 중요한 오해 풀기 → `push` ≠ `merge`**
- push = *같은 브랜치끼리* 업로드 (feat → origin/feat)
- merge = *서로 다른 브랜치* 결합 (feat → main)

---

## 1. 솔로 작업 vs 팀 작업 플로우

```
[팀]   각자 브랜치 → commit → push → PR(원격 머지) → production 브랜치 머지 → 배포
[솔로] main 에서      commit → push ───────────(끝)
```

### 솔로(= 지금 내 ImHereForU)
- `main` 하나에서 직접 작업 → `commit` → `push`
- 합칠 다른 브랜치가 없으니 **merge 단계가 아예 없음** (자동인 게 아니라 불필요한 것)

### 팀
- 여러 명이 **각자 다른 브랜치**에서 작업
- 그래서 `main`에 합치는 **merge(PR)** 가 필수
- 합치기 전에 **코드 리뷰(검토)** 를 끼워넣음

> 혼자면 합칠 사람도 리뷰할 사람도 없어서 머지·PR이 생략된다.

## 2. 브랜치에서 작업할 때 push와 merge

```
feat 브랜치에서 작업 → commit → push
                              ↓
                origin/feat 브랜치에 올라감  (main 아님!)
                              ↓
                merge: 그 브랜치를 main 에 합침
```

- **push는 "같은 이름 브랜치끼리"** 올라간다. `feat`에서 push하면 원격 `origin/feat`에 올라가고, **`main`은 안 건드린다.**
- 그 작업물을 `main`에 반영하려면 → **merge**가 따로 필요하다.

### merge 대상은 꼭 main만은 아니다
보통은 `main`이지만 팀마다 다르다.
예) `main`(또는 `develop`) → **production 브랜치**로 한 번 더 merge 후 배포.
→ merge는 "A를 B에 합치기"라는 일반 동작이고, B가 main일 때가 가장 흔할 뿐.

### merge를 "어디서" 하나
- **GitHub에서**: `feat` push하면 "Compare & Pull Request" 버튼 → PR 생성 → "Merge" 클릭 (팀 표준, 리뷰 끼우기 좋음)
- **로컬에서도 가능**: `git checkout main` → `git merge feat` (혼자일 땐 이렇게도)

## 3. 내 repo를 clone한 사람은 무엇을 보나

**GitHub(origin)에 push된 것까지만** 보인다.

- ✅ `origin/main`에 올라간 커밋 전부 (= GitHub에 올라간 모든 코드)
- ❌ push 안 한 로컬 전용 브랜치 (예: `feat/v3-character-overlay`) → GitHub에 없으니 안 보임
- ❌ 아직 commit 안 한 로컬 변경사항 → 당연히 안 보임

> "clone하면 보이는 것" = **GitHub에 push된 브랜치들.**

---

## 4. GitHub에 올라감 ≠ 배포됨

- **GitHub에 올라감** = 코드를 저장·공유한 상태
- **배포(deploy)** = 그 코드를 빌드해서 실제 서비스로 띄우는 것 (서버, 앱스토어 등)

→ GitHub에 있다고 배포된 게 아니다. 둘은 완전히 별개의 단계.

---

## 5. 면접에서 나오면

- **"push와 merge의 차이는?"**
  → push는 같은 브랜치 업로드, merge는 서로 다른 브랜치 결합
- **"PR(Pull Request)은 왜 쓰나?"**
  → 머지 전에 코드 리뷰/검증을 강제하는 협업 장치
- **"main에 직접 push vs 브랜치 전략?"**
  → 솔로/작은 변경은 trunk 직접, 팀/큰 변경은 브랜치+PR (git-flow, trunk-based 등)

---

## 한 문장 요약

> **commit**은 로컬 저장, **push**는 같은 브랜치 업로드, **merge**는 브랜치 합치기, **배포**는 서비스로 띄우기 — 네 개는 전부 다른 단계다.
